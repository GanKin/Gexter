# Plan 02-01: SSE Chat Endpoint + Session Registry

## Objective

建立从浏览器到 Agent 的完整流式通信管道：创建会话、注册到服务端、通过 SSE 端点消费 Agent 事件流。

## must_haves

- [ ] 浏览器可以 POST 一个问题到 SSE 端点，并收到流式 Agent 事件
- [ ] 会话创建后注册到服务端内存 Map，chat 端点能按 ID 找到会话
- [ ] 只有 CONTEXT.md 定义的 6 种事件被推送到前端
- [ ] SSE 响应格式正确（`text/event-stream`, `data: JSON\n\n`）

## Task 3: 创建 SSE Chat 端点

<read_first>
- src/webui/runtime/adapter.ts（runWebSession 适配器）
- src/webui/runtime/registry.ts（Session Registry）
- src/webui/runtime/types.ts（PHASE2_EVENT_TYPES, StreamableAgentEvent）
- src/agent/types.ts（AgentEvent 类型定义）
</read_first>

<action>
创建 `src/app/api/runtime/sessions/[id]/chat/route.ts`：

1. 设置 `export const dynamic = 'force-dynamic'`
2. POST handler 接收 `params: { id: string }`（Next.js 15 需要 `await params`）
3. 从 request body 解析 `{ query: string }`
4. 调用 `getSession(params.id)` 获取 session，不存在则返回 `404`
5. 检查 `session.status !== 'idle'` 时返回 `409 Conflict`（防止并发）
6. 创建 `ReadableStream`：
   - 使用 `new TextEncoder()` 编码
   - 定义 `sendEvent(event)` 函数：`controller.enqueue(encoder.encode(\`data: ${JSON.stringify(event)}\n\n\`))`
   - 调用 `runWebSession(session, { query, onEvent })` 
   - 在 `onEvent` 回调中过滤：`if (PHASE2_EVENT_TYPES.has(event.event.type))` 则调用 `sendEvent`
   - 使用 `try/catch`，错误时发送 `{ type: 'error', message: error.message }` 事件
   - 最后 `controller.close()`
7. 返回 `new Response(stream, { headers: { 'Content-Type': 'text/event-stream; charset=utf-8', 'Cache-Control': 'no-cache, no-store', 'X-Content-Type-Options': 'nosniff' } })`
8. 只支持 POST 方法，其他返回 405
</action>

<acceptance_criteria>
- `src/app/api/runtime/sessions/[id]/chat/route.ts` 存在
- 有 `export const dynamic = 'force-dynamic'`
- POST 请求返回 `Content-Type: text/event-stream; charset=utf-8`
- 不存在的 sessionId 返回 404
- 已在运行的 session 返回 409
- SSE 事件格式为 `data: {"type":"...","...":"..."}\n\n`
- 只有 `thinking`, `stream_progress`, `tool_start`, `tool_end`, `tool_error`, `done` 事件被推送
- 错误时发送 `{ type: 'error', message: '...' }` 事件并关闭流
</acceptance_criteria>
