- src/webui/runtime/types.ts（类型定义）
- src/lib/session-index.ts（会话索引接口，了解 SessionSummary 类型）
</read_first>

<action>
1. 创建 `src/__tests__/api-sessions.test.ts`：

2. 测试用例设计：

   **测试 1: POST /api/runtime/sessions 返回新会话**
   - 调用 `POST /api/runtime/sessions`（body 可以是空 `{}`）
   - 断言响应 status 为 200
   - 断言返回 JSON 包含 `sessionId` 字段（string 类型，非空）

   - 使用 `bun:test` 的 `describe` / `it` / `expect`

### Task 2: 创建 SSE 流式响应集成测试

<read_first>
- src/app/api/runtime/sessions/[id]/chat/route.ts（SSE chat handler）
- src/hooks/use-sse-stream.ts（前端 SSE 消费逻辑和事件类型）
- src/agent/types.ts（StreamProgressEvent、ToolStartEvent 等类型定义）
</read_first>

<action>
1. 创建 `src/__tests__/api-chat-stream.test.ts`：

2. 测试用例设计：

   **测试 1: POST /api/runtime/sessions/:id/chat 返回 SSE 流**
   - 先创建一个会话获得 sessionId
   - 构造 `POST /api/runtime/sessions/{id}/chat` 请求，body 为 `{ query: "test" }`
   - 断言响应 status 为 200
   - 断言响应 header 包含 `text/event-stream` content-type
   - 消费 SSE 流，断言至少收到一个 `message` 或 `done` 事件

   **测试 2: SSE 事件类型验证**
   - 在流中检查事件格式是否正确（`data: {...}\n\n`）
   - 断言 JSON 可解析且包含 `type` 字段

   **测试 3: 无效 sessionId 返回错误**
   - 构造 `POST /api/runtime/sessions/non-existent-id/chat` 请求
   - 断言响应 status 为 404 或 400

3. Mock 策略：
   - Mock LLM 调用：在测试中 mock `src/model/llm.ts` 的 `createChatCompletion` 或类似函数，返回固定的 SSE 数据
   - 或者使用 `src/agent/agent.ts` 的 agent runner，mock 掉实际 LLM 调用
   - 目的是验证 SSE 管道（route → event stream → response）不验证 LLM 质量
</action>
