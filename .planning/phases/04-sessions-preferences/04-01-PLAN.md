## Tasks

### Task 1: IndexedDB 会话存储层
**type:** implementation

创建 `src/lib/session-store.ts`，封装 IndexedDB 操作：

1. **Schema 设计:**
   - Store: `sessions` — 存储 sessionId, title, createdAt, lastActiveAt, model, messageCount
   - Store: `messages` — 存储 messageId, sessionId (索引), role, content, status, toolCalls (序列化 JSON), approvalRequest
   - 索引: `messages.sessionId` 用于按会话查询消息

2. **API 封装 (全部 Promise 化):**
   ```
   initDB() → Promise<IDBDatabase>
   saveSession(sessionId, metadata) → Promise<void>
   saveMessages(sessionId, messages[]) → Promise<void>
   loadSession(sessionId) → Promise<SessionMetadata | null>
   loadMessages(sessionId) → Promise<ChatMessage[]>
   listSessions() → Promise<SessionMetadata[]>  // 按 lastActiveAt 降序
   deleteSession(sessionId) → Promise<void>
   ```

3. **序列化规则:**
   - `toolCalls` 数组整体 JSON.stringify 后存入 IndexedDB
   - `approvalRequest` 同理 JSON.stringify
   - 读取时 JSON.parse 还原
   - 恢复时清除瞬态：`status` 非 complete 的消息设为 complete，thinking=false，thinkingMessage=null

4. **清理策略:**
   - 最多保留 50 个会话
   - 超出时删除最旧的（按 lastActiveAt）
   - 单个会话消息上限 500 条，超出时只保留最近 500 条

**验证:** 单独运行 `bun test src/lib/session-store.test.ts` 通过

**Files to create/modify:**
- `src/lib/session-store.ts` (新建)
- `src/lib/session-store.test.ts` (新建)

### Task 2: 会话元数据同步 — localStorage 索引
**type:** implementation

创建 `src/lib/session-index.ts`，管理 localStorage 中的会话索引：

1. **localStorage 结构:**
   ```
   key: "dexter-sessions"
   value: JSON.stringify([{
     sessionId: "web-xxx",
     title: "分析 AAPL 最新财报...",  // 首条用户消息前 30 字符
     createdAt: "2026-05-05T10:00:00Z",
     lastActiveAt: "2026-05-05T10:15:00Z",
     model: "gpt-5.4",
     messageCount: 6
   }, ...])
   ```

2. **API:**
   ```
   getSessionIndex() → SessionSummary[]
   addToIndex(sessionId, firstMessage, model) → void
   updateIndex(sessionId, {lastActiveAt?, messageCount?}) → void
   removeFromIndex(sessionId) → void
   getActiveSessionId() → string | null
   setActiveSessionId(sessionId) → void
   ```

3. **与 IndexedDB 协同:**
   - 新建会话时：addToIndex + IndexedDB saveSession
   - 发送消息时：updateIndex(lastActiveAt, messageCount) + IndexedDB saveMessages
   - 删除会话时：removeFromIndex + IndexedDB deleteSession
   - 初始化时：从 localStorage 读取索引渲染列表，消息内容按需从 IndexedDB 加载

**验证:** 索引操作正确，getActiveSessionId/setActiveSessionId 正常工作

**Files to create/modify:**
- `src/lib/session-index.ts` (新建)

### Task 3: 改造 useChatSession hook — 支持多会话切换
**type:** implementation

扩展 `src/hooks/use-chat-session.ts`：

1. **新增状态:**
   ```typescript
   const [sessionList, setSessionList] = useState<SessionSummary[]>([]);
   ```

2. **新增方法:**
   - `loadSessionList()` — 从 localStorage 索引读取会话列表，设置 sessionList 状态
   - `switchSession(targetSessionId: string)` — 核心方法：
     a. 保存当前会话消息到 IndexedDB（如果有）
     b. 从 IndexedDB 加载目标会话消息
     c. 清除瞬态（D-11），渲染历史消息
     d. 更新 sessionId 和 activeSessionId
     e. 在服务端创建新 session（如果目标 sessionId 对应的服务端 session 已不存在）
   - `startNewSession()` — 清空聊天区，重置 messages，准备新会话
   - `deleteSessionById(sessionId: string)` — 从 IndexedDB + localStorage 删除

3. **修改现有逻辑:**
   - `sendQuery` 完成后：同步消息到 IndexedDB + 更新 localStorage 索引
   - `createNewSession` 时：自动调用 addToIndex
   - 初始化时：调用 loadSessionList()

4. **服务端 session 重建:**
   - switchSession 后，如果需要继续对话，前端正常发 POST /sessions 创建新服务端 session
   - 历史消息不注入服务端 InMemoryChatHistory（前端展示层恢复即可，新对话上下文由 Agent 自行管理）

**验证:** 创建 2 个会话 → 各发 1 条消息 → 切换 → 确认消息不串台 → 刷新页面 → 恢复会话

## Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| IndexedDB 异步初始化时序问题 | Medium | initDB() 单例模式，所有操作 await initDB() |
| localStorage 容量溢出 | Low | 索引数据极小（每会话 <200 字节），50 会话 <10KB |
| 消息序列化丢失复杂类型 | Medium | toolCalls/approvalRequest 用 JSON.stringify/parse，测试覆盖 |
| 服务端 session 丢失后恢复失败 | Low | switchSession 检测服务端 session，不存在则创建新的 |
